Pure CSS Hamburger Menu

How to make a hamburger menu with pure CSS (no preprocessors)

Logic behind the setup

View the full HTML and CSS example on CodePen.

As I perused the internet for answers as to how to make this hamburger menu, I learned that I needed some sort of input in the HTML, like radio or checkbox. When the user provided the input (i.e. clicked on the box), it would then trigger the menu to drop down. However, we don’t want the user to actually see the checkbox. But how can the user click on it to trigger the menu dropdown if they can’t see it? I learned that we can give a label tag associated with the input. If the user clicks on the content within the label tag, it acts the same as if you had clicked on the input field itself. Pretty neat! In order for this to work, label needs a for attribute whose value matches the id attribute of your input. For example:

<input type=“checkbox” id=“menu-btn” />
<label for=“menu-btn”>
</label>

Great, so we know what we need in the HTML that we can style with CSS to trigger the menu to dropdown! Just one problem though. I want my logo (or h1 tag) and menu to be on the same line with each other. And then when the screen is smaller, I want the hamburger menu to be in the spot that the nav was just in and then have the nav drop down below them. How do we structure the whole header HTML to make that possible?? This was one of the biggest struggles I had when trying to figure this out!

What if…what if we used display: flex; for the header when I don’t want the hamburger menu and used display: grid; when I do want the hamburger menu?Let me explain. If my HTML is structured like this:

<header>
    <h1><a href="index.html">Logo here</a></h1>
                                
    <input class="menu-btn" type="checkbox" id="menu-btn" />
    <label class="menu-icon" for="menu-btn">
        <span class="navicon"></span>
    </label>

    <nav class=“menu">
        <a class=“nav-item” href="aboutme.html">About Me</a>
        <a class=“nav-item” href="blog.html">Blog</a>
        <a class=“nav-item” href="#">Resume</a>
    </nav>	
</header>

My h1, input, label, and nav elements are all siblings of header. So when I make the header a flex container, all of those siblings are now flex items. This allows me to set the flex-flow: row wrap; so that they are all on the same row together. Great! We’re one step closer. Now we need to find a way to hide the input and label so the user can’t actually see them. For this, we can do the following in our CSS:

#menu-btn {
    position: absolute;
    top: -100%;
    left: -100%;
}

Ok, so what’s going on here? #menu-btn is the id we gave our input element. What does position: absolute; do? Let’s quickly go over the position property!

The default for position is static. This means that the HTML elements are going to display in the browser where they would naturally “fall”, so to speak.

position: relative; allows you to change where your element is relative to where it normally would be if its position were static. So if you were to do top: 5px; for an element that has position: relative;, the element would be moved down 5px from the top of where that element would have been if you had left its position as static.

position: absolute; allows you to change where your element is relative to its closest ancestor whose position is not static. If the immediate parent just has default static positioning, but the grandparent has something else like position: relative;, position: fixed;, or position: sticky;, then your element will be moved relative to the edges of that non-static, or positioned, element. If none of its ancestor containing blocks are positioned, then your element is moved to be positioned relative to the edges of the initial containing block (usually the body). The Mozilla Web Docs have a nice little demo to see the positioning differences.

<div class=“grandparent”>
        <section class=“parent”>
            <div class=“child">
            </div>
        </section>
    </div>
    
.grandparent {
        position: relative;
    }
    
    .parent {
        position: static; 
    }
    
    .child {
        position: absolute;
    }

Now that we understand better what position: absolute; does, let’s look at our styling for our input element again:

#menu-btn {
    position: absolute;
    top: -100%;
    left: -100%;
}

Alright. So we know that #menu-btn, our input, is a child of our header. I never changed the position of the header, so it is just at the static default. Going back one more ancestor, we might reach the body element. Great! That would be our initial containing block, so that will be the element by which we determine where our #menu-btn goes!

For positioning, our options are top, bottom, left, and right. And the value that we give to those properties moves the element that far FROM that property. For example, we gave our #menu-btn a positioning of top: -100%; and left: -100%;. This will push our #menu-btn -100% of the size of the body from the top. In other words, towards the bottom. But here, since it is a negative value, it will go the opposite direction. Same thing with left: -100%;. Our #menu-btn will be pushed -100% from the left (towards the right) -100%. But again, since it is a negative value, it will end up moving left. Take a peek at that Mozilla demo if you’re still confused.

Wait, wouldn’t positioning the input #menu-btn like that push it off the screen? Exactly! Now no one will see it, but it is still there and can still be affected by the clicks of the user on the label. Ta-da! Easy peasy, as Bob Newby would say.

Ok, so now that we’ve successfully “hidden” our input #menu-btn, now we need to make our label invisible. We can do this by simply setting visibility: hidden;.

Great, so now all we can see on our header is the logo and the nav! Yay! We know we can use a media query to change the styling when we have a different size screen, but what would we do? We didn’t actually do much styling yet. And where’s this hamburger icon you talked about? We don’t have it yet! We haven’t needed it until we reached this media query, right? (Assuming you started with a desktop-first approach and are doing a max-width media query for smaller screens). Buckle up, kids. This is where the fun begins :)

Let’s revisit our HTML structure a bit. We have everything in our header as siblings, right? We set display: flex; and flex-flow: row wrap; to make the logo, input, label, and nav all on the same row. But wait. For a smaller screen, we just want the logo and the hamburger icon on the same row. We want the nav to show up one row down! Oh no, we structured our HTML wrong! Nope! This is where we can work some CSS grid magic :) If you’re not familiar with CSS grids, I highly recommend the tutorial by Wes Bos. He does a great job and it’s free!

Assuming you know a bit about CSS grid, we’re going to use that for our header inside of our media query for smaller screens. So we’re going to override the display: flex; we had before in our header and now give it display: grid;. Why are we using grid, you ask? Well, not counting our input #menu-btn that we’ve positioned off screen, we have 3 items in our header: the logo, the label, and the nav. But now we just want the logo and the hamburger icon on one row and the nav on another row. This is where grid comes in. We can now give the header grid-template-areas to create 3 columns and 2 rows for our content, while simultaneously naming them:

header {
    grid-template-areas:
    “logo	logo	hamburger”
    “nav	nav     nav”;
}

Assuming you want your in the lefthand corner and the hamburger icon in the righthand corner, we can do something like this with our grid areas. Then all we have to do is call the selectors for the logo, label, and nav and give them their corresponding grid-area names and CSS grid works its magic and places them there!

Making the hamburger icon

Alrighty, let’s talk about this hamburger icon. Finally. Something that I excluded from the HTML snippets up until now is this little span that nests inside of our label element. Giving the label a child that spans the label makes our clickable input area, so to speak, larger so that it will trigger our dropdown menu.

<header>
        <h1>Logo here</h1>
        <input class=“menu-btn” type=“checkbox” id=“menu-btn” />
        <label class=“menu-icon” for=“menu-btn”>
            <span class=“navicon”></span>
        </label>
        <nav class=“menu”>
            <a class=“nav-item” href=“”>Home</a>
            <a class=“nav-item” href=“”>About</a>
            <a class=“nav-item” href=“”>Contact</a>
        </nav>
    </header>

Now let’s give the span some styling so it looks like a hamburger icon! We can give it a background-color of whatever color we want our icon to be. Then we’ll give it display: block; and set the width and height. The lines on hamburger menus are pretty small, so we our sizings will be small. You might notice that only 1 line shows up. But hamburger menus usually have at least 2, if not 3! How do we make the others? This is where :before and :after pseudo-classes come in. :before let’s you create a “duplicate” of the selector and position it so that the browser displays the duplicate before the original, as if you had stuck another in the HTML right before the original. :after works in the same way, but instead of coming before the original, the browser will display it after. Pretty straightforward.

But without giving it any positioning and since our span .navicon is so small, those pseudo-classes just sit on top of the original and you can’t see a difference! No worries, we can add some positioning! For example:

header .navicon:before {
        top: 5px;
    }

    header .navicon:after {
        bottom: 5px;
    }

Styling our pseudo-classes like this will create 2 duplicate “lines” identical to the original span .navicon and position them 5px above and below it. Ta-da! A 3-line hamburger icon all in CSS. Sweet.

Now that we have our cute burger, let’s make it have a smooth transition when we click on it. After all, since it’s nested inside of our label, clicking the hamburger will provide the check for our input. So let’s make it do something!

We can add a transition attribute to our span .navicon and its :before and :after pseudo-classes.

Then let’s make sure our nav is hiding. We want to make sure we can’t see it while the hamburger isn’t clicked. We can set max-height: 0; and have overflow: hidden;. Makes sense, right? If the height is zero, we won’t be able to see it the nav. So when we click on our hamburger, we need to override that height so that the nav will show up!

header .menu-btn:checked ~ .menu {
        max-height: 250px;
    }

If we want to be a little bit fancy and have the hamburger icon turn into an X when the nav drops down (assuming we already gave the span transition properties earlier), we can now apply the following styling:

header .menu-btn:checked ~ .menu-icon .navicon {
        background: transparent;
    }

So what is this saying? When our input .menu-btn (via the label) is checked, give the span .navicon a background: transparent;. Note that this only gives the original one a transparent background and not the :before and :after lines that were created from it. Now we should be left with the top and bottom lines of the hamburger icon.

Now to make them form an X! We have 2 parallel lines, so essentially all we need them to do is rotate 45 degrees. We can do that with the transform property.

header .menu-btn:checked ~ .menu-icon .navicon:before {
        transform: rotate(-45deg);
    }
    
    header .menu-btn:checked ~ .menu-icon .navicon:after {
        transform: rotate(45deg);
    }

And then we can just tweak the positioning a little bit and voila! An X!

header .menu-btn:checked ~ .menu-icon:not(.steps) .navicon:before,
header .menu-btn:checked ~ .menu-icon:not(.steps) .navicon:after {
    top: 0;
}

Side note, we can give our label .menu-icon some padding so that the clickable area is a bit bigger, making it easier for the user to trigger the dropdown menu.
